## Sintesi

CutsceneScreen è ora **dual-mode**: distingue tra una cutscene scritta in Ink (renderizzata tramite NarrativeService + narrativeStore) e una cutscene legacy a pagine (renderizzata col vecchio CutsceneDatabase). La scelta avviene a monte, in `gameStore.startCutscene(id)`, sulla base di una mappa esplicita `LEGACY_CUTSCENE_TO_INK_KNOT`.

Approccio motivato dal fatto che solo **l'intro è già scritta in Ink** (`intro` knot, 138 righe, completa). Le altre ~9 cutscene legacy resteranno tali finché non vengono riscritte in Ink — lavoro contenutistico separato dall'architettura.

## Modifiche tecniche

1. **NarrativeService**: nuovo metodo `startCutscene(knot, returnState?)`. Differenza chiave con `startDialogue`: default returnState = `IN_GAME` (le cutscene sono transizioni di scena, non interlude). Setta `GameState.CUTSCENE` e fa `jumpTo`.

2. **gameStore.startCutscene(id)**: ora consulta `LEGACY_CUTSCENE_TO_INK_KNOT` (top-level const) e, se trova un mapping, instrada al NarrativeService. Altrimenti percorso legacy invariato. Aggiungere un nuovo mapping = aggiungere una riga a quel record.

3. **CutsceneScreen.tsx**: refactor in due sotto-componenti.
   - `InkCutscene`: legge `currentText` e `currentChoices` dal narrativeStore. Split del testo per `\n\n`, reveal a paragrafo (1s/paragrafo). INVIO mostra-tutto / continua / sceglie. ESC = endDialogue. Le scelte [1-9] vanno a `narrativeService.chooseChoiceIndex`.
   - `LegacyCutscene`: comportamento originale invariato (page-based con `activeCutscene`).
   - Root sceglie tra i due in base a `narrativeStore.isStoryActive`.
